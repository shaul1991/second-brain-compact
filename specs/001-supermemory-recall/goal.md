# Goal: second-brain-compact 에 시맨틱 검색·기억 엔진(supermemory) 결합

## Background (배경)
second-brain-compact 는 로컬 Markdown 볼트 기반의 private-only 세컨드 브레인이다.
메모는 "임시(capture) → 사람이 승격(promote) → 정식 노트(notes)" 흐름을 따르며,
사람이 승격하기 전까지는 검색에 노출되지 않는다. 거버넌스·소유권·provenance 는
이 모델로 충분히 확보되어 있다.

그러나 검색(recall)은 글자 일치 기반의 키워드 검색뿐이며(시맨틱 점수는 0 으로 비어 있는
스텁 상태), 매 호출마다 볼트 전체를 훑는다. supermemory 는 로컬 실행이 가능한
검색·기억 엔진으로, 의미 기반 검색과 메모리 추출에서 검증된 도구다.

또한 이 시스템은 Claude·Codex·Gemini 등에서 공통으로 꽂아 쓰는 MCP 플러그인 형태로
이미 설계되어 있다(README 의 등록 예시). 사용자는 이를 git·docker 처럼 "한 번 설치해
늘 쓰는 기본 플러그인"으로 활용하고자 한다.

## Problem (문제)
- 검색이 글자 일치에만 의존해 "뜻으로" 회상할 수 없다. 볼트가 커질수록 비용도 커진다.
- 쌓이는 임시 메모(capture backlog)를 사람이 일일이 분류해야 하며, 정리를 보조하는
  지능이 없다.
- 요약하면, 이 시스템은 "기록·거버넌스"는 충분하나 "회상·정리 지능"이 비어 있다.

## Objective (목표)
compact 의 거버넌스·소유·privacy 모델을 **그대로 유지한 채**, 검색을 의미 기반으로
끌어올리고, 임시 메모의 정리를 비서(supermemory)가 **제안**으로 보조하게 한다.
AI 도구는 여전히 단일 MCP 플러그인 하나만 꽂으며, supermemory 는 그 뒤에 숨는다.

핵심 불변식 두 가지를 깨지 않는다:
1. **승격(도장)은 사람만 한다.** supermemory 는 제안만 하고 결정하지 않는다.
2. **모든 것은 로컬에서 동작한다.** 메모는 외부(클라우드)로 나가지 않는다.

## Expected outcome (기대 결과)
- 글자가 똑같지 않아도 의미가 통하면 노트를 찾을 수 있다.
- "이 메모들 묶을까요?" / "이 노트가 기존 노트와 모순됩니다" 같은 정리 제안이
  임시 서랍으로 들어와, 사람이 승격/폐기를 결정한다.
- Markdown 이 여전히 단일 정본이며, supermemory 는 언제든 볼트에서 다시 만들 수 있는
  파생 인덱스로만 존재한다.
- Claude·Codex·Gemini 어디서나 동일한 단일 플러그인으로 접속하며, 검색 품질만
  향상된다.

## Success metrics (성공 지표)
- 키워드가 정확히 일치하지 않는 질의에서도 관련 노트를 회상한다.
- privacy 위반 0: restricted 메모가 recall 결과로 새지 않는다 / 외부 전송 0 건.
- supermemory 를 제거해도 볼트는 무손상이며, 인덱스는 볼트로부터 재구축된다.
- 새 기기에서 볼트(Markdown)만 받아오면 인덱스를 재구축해 동일하게 동작한다.
- 임시 메모 분류에 드는 사람의 수작업이 제안으로 줄어든다.

## Non-goals (이번 범위 제외)
- 클라우드 호스팅 supermemory 사용.
- supermemory 에 쓰기·승격·삭제 권한 부여.
- Markdown 정본 모델 교체 (supermemory 를 진실의 원천으로 삼지 않는다).
- 사람 없는 자동 승격.
- Obsidian 적극 연동(관계 시각화·그래프 강화 등)은 별도 goal 로 다룬다.
  이번에는 "Obsidian 으로 열 수 있는 상태를 깨지 않는다"는 제약만 둔다.

## Constraints (제약)
- 전부 로컬·오프라인 동작. 인터넷 전송 금지.
- 사람-루프 승격 규칙 보존.
- Markdown = 단일 정본, supermemory = 파생.
- 노트는 평범한 Markdown 으로 유지되어 Obsidian 등 외부 뷰어로 열 수 있어야 한다.
- 백업·기기 이전이 사용자 소유 git 원격으로 계속 가능해야 하며, 파생 인덱스는
  새 기기에서 볼트로부터 재구축 가능해야 한다.
- AI 도구는 여전히 단일 second-brain MCP 플러그인 하나만 꽂는다(supermemory 는 그 뒤).
- 기존 "외부 의존성 최소" 철학 존중 — supermemory 는 선택적(opt-in) 백엔드로 두고,
  키워드 검색 폴백을 유지한다.
- 모든 도구 호출의 감사 로깅을 유지한다.

## Stakeholders (이해관계자)
- 사용자(단독 소유자이자 운영자).
- AI 비서(검색·제안의 소비자).
- 향후 다른 기기로의 동기화.

## Risks (리스크)
- supermemory 가 정리 보조를 위해 임시 메모 본문을 보게 될 경우의 노출 위험 →
  로컬 격리 + recall 차단 칸막이가 전제.
- 파생 인덱스와 정본 사이의 드리프트.
- 외부 의존성 도입으로 compact 의 단순성이 훼손될 위험.
- 비서 제안 품질이 낮으면 임시 서랍에 노이즈가 쌓일 위험.

- 성공 지표의 정량화 방법.
